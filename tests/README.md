# Tests

Тут лежать короткі й вузькі перевірки, щоб після змін було видно, що саме ми не зламали.

## Як запускати

Усі тести:

```bash
python -m unittest discover -s tests -v
```

Один файл:

```bash
python -m unittest tests.test_deadline_policy -v
```

Один конкретний тест:

```bash
python -m unittest tests.test_deadline_policy.DeadlinePolicyTests.test_per_user_override_is_used_for_due_at -v
```

## Що де перевіряється

- `test_logic.py`
  Перевіряє стабільність базового порядку чергувань із `residents.json`.
- `test_scheduler.py`
  Перевіряє базові правила дедлайнів scheduler-а.
- `test_deadline_policy.py`
  Перевіряє нові правила навколо персональних дедлайнів, skip-date і календарних винятків.
- `test_backup_service.py`
  Перевіряє відновлення з бекапу.
- `test_export.py`
  Перевіряє порядок людей в експорті.
- `test_database_integration.py`
  Окремий інтеграційний тест для SQLite. Він може бути `skipped`, якщо середовище не підходить.

## Як ці тести працюють

- Тут переважно `unit`-тести: вони не піднімають усього бота.
- Для простих сценаріїв використовується маленька async-заглушка `_FakeDb`, яка просто повертає значення з пам'яті.
- Там, де треба ізолювати одну гілку логіки, використовується `unittest.mock.patch`.
  Наприклад, у `test_deadline_policy.py` ми підміняємо `get_user_report_zones(...)`, щоб перевірити саме дедлайн-логіку, а не весь графік чергувань одразу.

## Коли додавати новий тест

Додавай тест, якщо:

- міняється дедлайн або scheduler-логіка;
- додається новий адмінський виняток;
- змінюється правило, яке залежить від дати, часу або timezone;
- виправляється баг, який уже відловили в логах.

Найкраще правило: один баг з продакшну -> один окремий тест поруч.
